# 缠论 C99 内存设计

## 一、架构概览

```
┌──────────────────────────────────────────────────────────────┐
│                        观察者 (heap)                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐       ┌──────────┐  │
│  │ 普K序列   │ │ 缠K序列   │ │ 分型序列  │  ...  │ 中枢序列  │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘       └────┬─────┘  │
│       │             │             │                  │        │
│       │    弱引用    │    弱引用    │   弱引用          │  弱引用 │
│       ▼             ▼             ▼                  ▼        │
│  ┌──────────────────────────────────────────────────────┐    │
│  │              全局内存池 (单例)                          │    │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │    │
│  │  │ K线   │ │ 缠K   │ │ 分型  │ │  笔   │ │ 中枢  │  ... │    │
│  │  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘       │    │
│  │        bump-pointer 分配 · 清理链表 · 统一释放          │    │
│  └──────────────────────────────────────────────────────┘    │
│                                                               │
│  配置 (强引用例外) ──── 引用()/解引用() → free()              │
└──────────────────────────────────────────────────────────────┘
```

**核心原则：全局内存池是所有对象的唯一所有者。** 所有对象间引用均为弱引用——只追踪引用关系，不控制生命周期。对象的 struct 内存由池分配，池释放时统一回收，绝不单独 `free()`。

唯一例外是**观察者本身**和**观察者→配置**引用，它们使用传统强引用（引用计数控制释放），因为观察者是堆分配的，不在池内。

## 二、内存池 (Arena Allocator)

### 2.1 数据结构 (`arena.h`)

```
内存池
├── 首块 → 内存池块 (64KB~16MB)
│           ├── 下一块 → 内存池块
│           │             ├── 下一块 → ...
│           │             ├── 容量 / 已用 (bump offset)
│           │             └── 数据区 [...对象...]
│           ├── 容量 / 已用
│           └── 数据区 [...对象...]
├── 当前块 (bump-pointer 目标)
├── 块大小 / 总容量 / 总分配 / 分配次数
└── 清理头 → 对象A → 对象B → 对象C → ... (侵入式链表)
```

### 2.2 分配流程

```
内存池_分配(pool, size)
  │
  ├── 当前块剩余 ≥ size → bump pointer (两条指令，~5ns)
  │
  └── 当前块剩余 < size
        ├── size ≤ 块大小 → 分配新块 (2x 扩容)，bump from new block
        └── size > 块大小 → 分配 oversized 专用块
```

块大小选择：
| 初始块大小 | 适用场景 |
|---|---|
| 64 KB | 小型数据 (< 5000 根K线) |
| 1 MB (默认) | 中等数据 (~10000 根) |
| 16 MB | 大型数据 (~50000 根，推荐) |

### 2.3 全局池单例

```c
static 内存池 *获取全局池(void) {
    static 内存池 全局池;
    static bool 已初始化 = false;
    if (!已初始化) {
        内存池_初始化(&全局池, 0);  // 默认 1MB 块
        已初始化 = true;
    }
    return &全局池;
}
```

全局池在首次 `分配()` 调用时自动创建，程序运行期间不释放（单例模式）。通过 `释放全局内存池()` 手动释放。

## 三、对象头

每个池分配的对象以 `对象头结构` 开头（`chan.h:49-56`）：

```c
typedef struct {
    int       引用计数;      // pool=1(所有权标记), heap=正常计数
    对象类型  类型标记;      // 12 种类型枚举
    内存池    *所属内存池;   // NULL=堆分配, 非NULL=池分配
    void      *下一清理对象; // 侵入式链表，串联所有池对象
    bool      已销毁;        // 对象销毁() 已调用，防止重复销毁
    int       弱引用计数;    // 指向本对象的所有弱引用数量
} 对象头结构;
```

**两个计数字段的区别：**

| 字段 | 池对象初始值 | 含义 | 控制什么 |
|---|---|---|---|
| `引用计数` | 1 | 所有权标记，永不为 0 | 池对象不因解引用而释放 |
| `弱引用计数` | 0 | 弱引用追踪，动态增减 | 验证引用一致性，assert(==0) 才能销毁 |

## 四、分配流程 (`分配()`)

```
分配(大小, 类型)
  │
  ├── 全局内存池 不存在 → 获取全局池() 自动创建
  │
  ├── 内存池_分配(pool, 大小)  ← bump-pointer，零初始化
  │
  ├── 初始化对象头:
  │     引用计数     = 1      (池所有权标记)
  │     类型标记     = 类型
  │     所属内存池   = pool
  │     弱引用计数   = 0
  │     下一清理对象 = pool->清理头   ← 串入清理链表
  │     pool->清理头 = ptr
  │
  └── 分配计数[类型]++
```

**关键点：**
- 池对象 `引用计数=1` 永不归零 → `解引用` 不会触发 `free()`
- 对象通过 `下一清理对象` 串成侵入式链表，释放时遍历
- 弱引用计数从 0 开始，随引用关系建立/解除而增减

## 五、弱引用模型

### 5.1 为什么全弱引用

旧模型（引用计数控制生命周期）存在两个问题：
1. **引用环**：虚线↔中枢、虚线↔线段特征之间形成循环引用，需要回收站 + 三阶段清理来打破
2. **空转开销**：全局池已持有所有对象内存，引用计数的增减实际上不影响释放时机

全弱引用化后：全局池是唯一所有者，对象间引用仅追踪关系。释放时遍历清理链表直接销毁，无需回收站。

### 5.2 弱引用操作

#### 独立指针字段 —— `弱引用_设置(ptr, field, target)`

```c
#define 弱引用_设置(ptr, field, target) do { \
    if ((ptr)->field && !已销毁((ptr)->field)) \
        ((对象头结构*)((ptr)->field))->弱引用计数--; \
    (ptr)->field = (target); \
    if (target) \
        ((对象头结构*)(target))->弱引用计数++; \
} while(0)
```

**用途：** 分型.左/中/右、虚线.文/武、中枢.第三买卖线 等独立指针字段。

**流程：** 旧目标弱引用计数-1 → 赋值 → 新目标弱引用计数+1

#### 动态数组 —— `弱引用_数组追加/设置/弹出/清除`

| 函数 | 等价操作 | 弱引用计数变化 |
|---|---|---|
| `弱引用_数组追加(arr, e)` | `动态数组_追加` + 弱引用++ | e +1 |
| `弱引用_数组设置(arr, i, e)` | 旧元素弱引用-- + `动态数组_设置` + e弱引用++ | 旧 -1, 新 +1 |
| `弱引用_数组弹出(arr)` | `动态数组_弹出` + 尾元素弱引用-- | 尾 -1 |
| `弱引用_数组清除(arr)` | 遍历弱引用-- + `free(数据)` + 归零 | 所有元素 -1 |

**用途：** 观察者序列、虚线.基础序列/特征序列、中枢.元素、线段特征.元素 等动态数组。

NULL 元素安全：所有弱引用函数在 `元素==NULL` 时不做任何操作。

#### 手动操作（极少数场景）

```c
弱引用_手动增加(ptr);  // ptr->弱引用计数++
弱引用_手动减少(ptr);  // ptr->弱引用计数--
```

仅在手动平衡复杂场景使用（如笔_弹出旧笔中的手动补偿）。

### 5.3 在代码中的使用模式

```c
// 模式 1: 序列追加（最常见）
弱引用_数组追加(分型序列, 当前分型);      // 替代旧的 动态数组_追加 + 引用()

// 模式 2: 序列弹出
分型 *旧分型 = 弱引用_数组弹出(分型序列);  // 替代旧的 动态数组_弹出 + 解引用()

// 模式 3: 独立字段赋值
弱引用_设置(段, 武, 武);                  // 替代旧的 段->武 = 武

// 模式 4: 清理时释放
弱引用_数组清除(&obj->基础序列);          // 替代旧的 动态数组_全部解引用()
弱引用_设置(obj, 前一缺口, NULL);          // 替代旧的 解引用(obj->前一缺口)
```

## 六、解引用与对象销毁

### 6.1 `解引用(对象)` —— 仅用于堆对象

```c
void 解引用(void *对象) {
    if (--引用计数(对象) <= 0) {
        对象销毁(对象);
        if (!((对象头结构*)对象)->所属内存池)
            free(对象);   // ← 仅堆分配对象执行 free
        // 池对象：struct 内存归池所有，不 free
    }
}
```

**当前唯一触发 free 的路径：** 观察者（calloc）和配置（池分配但走强引用）。

### 6.2 `对象销毁(对象)` —— 释放内部资源

```c
void 对象销毁(void *对象) {
    if (已销毁(对象)) return;      // 幂等
    已销毁(对象) = true;
    释放计数[类型]++;              // 每对象只销毁一次
    switch (类型) {
        case CHAN_TYPE_虚线:   释放虚线((虚线*)对象);
            // → 弱引用_数组清除(&基础序列)
            // → 弱引用_数组清除(&特征序列)
            // → 弱引用_数组清除(&实_中枢序列)
            // → 弱引用_数组清除(&虚_中枢序列)
            // → 弱引用_数组清除(&合_中枢序列)
            // → 弱引用_设置(前一缺口, NULL)
            // → 弱引用_设置(前一结束位置, NULL)
        ...
    }
}
```

**关键：** 各类型的 `释放*` 函数只清理内部弱引用（减计数 + free 内部 malloc 数组），不释放子对象 struct 本身。子对象由池统一管理。

## 七、观察者 —— 唯一的堆对象

```c
观察者 *观察者_新建(...) {
    观察者 *o = calloc(1, sizeof(观察者));  // 堆分配
    o->引用计数   = 1;
    o->所属内存池 = NULL;                    // NULL = 堆分配标记
    o->下一清理对象 = NULL;                  // 不在池清理链中
    ...
}
```

**为什么观察者不用池分配：** 它是顶层协调器，生命周期可能独立于单次分析。且它持有所有序列数组，如果放在池内，释放池时自身已经悬空。

**观察者→配置 是强引用：** 配置对象虽然在池内分配，但观察者使用 `引用(配置)` / `解引用(配置)` 管理。这是唯一保留的强引用路径，因为它简单且不形成循环。

## 八、释放流程（完整生命周期）

```
程序结束
  │
  ├── 1. 释放观察者(obs)
  │     │
  │     ├── Phase 1: 清理所有序列中的弱引用
  │     │    弱引用_数组清除(普通K线序列)
  │     │    弱引用_数组清除(缠论K线序列)
  │     │    弱引用_数组清除(基础缠K序列)
  │     │    弱引用_数组清除(分型序列)
  │     │    弱引用_数组清除(笔序列)
  │     │    弱引用_数组清除(笔_中枢序列)
  │     │    弱引用_数组清除(线段序列)
  │     │    弱引用_数组清除(中枢序列)
  │     │    弱引用_数组清除(扩展线段序列)
  │     │    弱引用_数组清除(扩展中枢序列)
  │     │    弱引用_数组清除(扩展线段序列_线段)
  │     │    弱引用_数组清除(扩展中枢序列_线段)
  │     │    弱引用_数组清除(线段_线段序列)
  │     │    弱引用_数组清除(线段_中枢序列)
  │     │    弱引用_数组清除(扩展线段序列_扩展线段)
  │     │    弱引用_数组清除(扩展中枢序列_扩展线段)
  │     │    → 所有 leaf 对象的弱引用计数归零
  │     │
  │     └── Phase 2: 释放配置（强引用）
  │          解引用(obs->配置);  → 配置.引用计数 0→对象销毁→free
  │
  ├── 2. 释放全局内存池()
  │     │
  │     ├── 遍历清理链表:
  │     │    for each obj in pool->清理头:
  │     │        if (!已销毁(obj)):
  │     │            assert(弱引用计数 == 0)  ← 安全网
  │     │            对象销毁(obj)              ← 清理内部 malloc 数组
  │     │
  │     └── 内存池_释放(pool):
  │           遍历所有内存池块 → free(块)
  │
  └── 3. 打印内存摘要
       分配计数 vs 释放计数 → 零泄漏确认
```

**为什么不需要回收站：** 旧模型中弹出/替换产生的孤立对象需要放入回收站，然后去重、打破循环释放。全弱引用后，孤立对象会自动归零弱引用计数（因为无人指向它），池释放时直接通过清理链表销毁即可。

## 九、验证系统

### 9.1 核心思路：独立重建 + 三方对比

```
                ┌──────────────┐
                │  快照存储值    │  ← 对象当前的弱引用计数值
                └──────┬───────┘
                       │ 对比
       ┌───────────────┼───────────────┐
       │               │               │
       ▼               ▼               │
  ┌─────────┐    ┌──────────┐         │
  │ 观察者    │    │ 池内交叉   │         │
  │ 序列引用  │ +  │ 引用重建  │ = 重建值 │
  └─────────┘    └──────────┘         │
       │               │               │
       ▼               ▼               ▼
  16 个 OBS_SEQ    遍历所有池对象    存储值 == 重建值 ?
  逐元素弱引用++   扫描内部指针/数组     ↓
                                    差异 = 0 ✓
```

### 9.2 执行步骤

```
验证弱引用计数(obs)
  │
  ├── 1. 收集快照
  │     遍历 pool->清理头 → snaps[n] = {ptr, 存储值}
  │     同时构建 idxs[n] 用于二分查找
  │
  ├── 2. 重置所有弱引用计数为 0
  │
  ├── 3a. 观察者视角重建
  │      遍历 16 个 OBS_SEQ 数组 → 逐元素弱引用++
  │      记录每个对象的 观察者贡献
  │      (obs->配置 是强引用，不纳入)
  │
  ├── 3b. 池内交叉引用重建
  │      遍历所有池对象:
  │        K线:         macd, rsi, kdj 指针
  │        分型:         左, 中, 右 指针
  │        虚线(笔/线段): 基础序列, 特征序列, 中枢序列, 文, 武, 前一缺口 等
  │        中枢:         元素, 第三买卖线
  │        线段特征:     元素
  │        特征分型:     左, 中, 右
  │      (缠论K线.标的K线 不参与——K线不走弱引用系统)
  │
  ├── 4. 三方对比
  │     for each 对象:
  │       存储 = snaps[i].存储值
  │       重建 = snaps[i].ptr->弱引用计数
  │       差值 = 存储 - 重建
  │       if 差值 != 0 → 打印差异行
  │
  └── 5. 恢复存储值
       所有对象弱引用计数恢复为原始值
```

### 9.3 验证结果解读

```
=== 验证弱引用计数: 池对象=859122 ===
  观察者序列引用: 858968 次
  池内交叉引用:   692877 次
  引用总计:       1551845 次

  结果: 全部 859122 对象弱引用计数一致 ✓
```

- **观察者序列引用**：16 个序列数组中元素的弱引用贡献

- **一致**：存储值与独立重建值完全匹配 → 所有弱引用操作都正确配对

## 十、动态数组的双层设计

### 10.1 底层：`动态数组_*` (chan.c)

原始动态数组操作，只管理 `void*` 缓冲区，不关心引用计数：

```
动态数组_追加  → realloc + 赋值
动态数组_弹出  → 返回尾元素，长度-1
动态数组_获取  → 下标访问
动态数组_设置  → 下标赋值
动态数组_释放  → (可选)遍历解引用 + free(数据)
```

### 10.2 弱引用层：`弱引用_数组*` (chan.c)

在底层之上包装弱引用计数操作。所有 C 内部代码使用此层：

```
弱引用_数组追加  → 动态数组_追加 + 元素->弱引用计数++
弱引用_数组弹出  → 动态数组_弹出 + 尾元素->弱引用计数--
弱引用_数组设置  → 旧元素-- + 动态数组_设置 + 新元素++
弱引用_数组清除  → 遍历减计数 + free(数据)
```

### 10.3 Python 层：`Py_动态数组_*` (_core.c)

Python `动态数组` 类型的 C 实现，**也已全部使用弱引用层**：

| Python 方法 | C 实现 | 弱引用计数 |
|---|---|---|
| `append(obj)` | `弱引用_数组追加(&arr, ptr)` | obj +1 |
| `pop()` | `弱引用_数组弹出(&arr)` | 尾元素 -1 |
| `pop_typed()` | `弱引用_数组弹出(&arr)` | 尾元素 -1 |
| `clear()` | `弱引用_数组清除(&arr)` + 重新初始化 | 全部元素 -1 |
| `tp_dealloc` | `弱引用_数组清除(&arr)` | 全部元素 -1 |

## 十一、关键不变项

1. **池对象绝不 free()** — `所属内存池 != NULL` → 解引用归零时 skip free
2. **对象销毁幂等** — `已销毁` 标志防止池清理时重复销毁
3. **弱引用计数对称** — 每处 `弱引用_数组追加` 必有对应的 `弱引用_数组弹出` 或 `弱引用_数组清除`
4. **观察者堆分配** — `所属内存池 == NULL` → 解引用归零时 `free()`
5. **配置强引用** — 观察者→配置走 `引用()/解引用()`，不纳入弱引用验证
6. **验证函数无副作用** — 先快照、重置、重建、对比、最后恢复原始值
7. **NULL 安全** — 所有弱引用函数对 NULL 输入无操作

## 十二、常见陷阱与修复记录

### 陷阱 1：直接指针赋值绕过弱引用

```c
// 错误：段->武 = 武;
// 正确：
弱引用_设置(段, 武, 武);
```

直接赋值导致旧分型弱引用计数 +1（幽灵引用），新分型弱引用计数 -1。修复位置：`线段_武斗` (chan.c:2144, 2159)。

### 陷阱 2：直接修改动态数组长度

```c
// 错误：self->元素.长度 = 有效长度;
// 正确：
while (self->元素.长度 > 有效长度)
    弱引用_数组弹出(&self->元素);
```

直接修改长度导致被截断元素的弱引用计数未减少。修复位置：`中枢_校验合法性`。

### 陷阱 3：局部动态数组未释放

```c
// 线段_刷新特征序列 中局部创建的 分析用序列 在函数结束时未释放
// 修复：函数末尾添加
动态数组_释放(&分析用序列, false);  // 仅释放数组内存，不操作元素弱引用
```

导致 ASan 报告 ~7MB 内存泄漏。

### 陷阱 4：Python 动态数组不使用弱引用层

`_core.c` 中 `Py_动态数组_追加/弹出/清空/释放` 最初使用原始 `动态数组_*`，不统计弱引用。Python 端创建动态数组并追加元素时，元素的弱引用计数不会变化。

修复：全部改为 `弱引用_数组追加/弹出/清除`。

### 陷阱 5：验证函数误纳入强引用

验证函数最初将 `obs->配置` 纳入弱引用统计，但配置是强引用。导致配置对象出现弱引用计数差异。

修复：在观察者重建、池内遍历、交叉检查三处跳过 `obs->配置`。
